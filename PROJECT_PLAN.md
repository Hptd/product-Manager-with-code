# AI 对话驱动的产品管理系统 - 项目规划

## 项目概述

基于浏览器实现一个**AI 对话驱动的原型制作系统**，通过内嵌终端调用 AI Code CLI 实时修改文件，将传统手动原型制作转变为"对话即设计"的交互模式。

### 核心价值主张

```
选择页面 → 选中 UI 元素 → 拖拽/缩放调整 → 输入修改指令 → AI 实时修改文件 → 预览更新
```

## 核心功能模块

### 1. 文件管理系统
- **多 HTML 页面管理**：支持创建、编辑、删除、重命名页面
- **资源管理**：图片、CSS、JS 文件的上传、组织、引用
- **页面导航树**：类似 Axure 的页面层级结构
- **页面跳转配置**：定义页面间的导航关系

### 2. 核心渲染 HTML 页面效果展示区
- **实时渲染引擎**：基于 iframe 或 Shadow DOM 隔离渲染
- **热更新支持**：文件修改后自动刷新预览
- **交互支持**：完整的 HTML/CSS/JS 交互能力
- **多页面切换**：在渲染区内切换不同页面

### 3. 页面 UI 选择系统（集成 ai-ui-runtime）
- **元素选中**：点击渲染区域内的任意元素
- **拖拽/缩放**：使用 interactjs 实现视觉调整
- **意图生成**：将操作转换为结构化 Intent JSON
- **多选支持**：Ctrl/Cmd 多选组件批量操作

### 4. AI 对话历史区域 + 用户对话框
- **对话历史**：展示与 AI 的完整交互记录
- **指令输入**：支持自然语言描述修改需求
- **上下文绑定**：将选中的 UI 元素意图自动附加到对话
- **代码预览**：展示 AI 生成的修改方案

### 5. Windows 终端集成
- **终端嵌入**：使用 xterm.js 或类似方案
- **CLI 调用**：执行 AI Code CLI 命令
- **实时反馈**：显示命令执行结果
- **文件监听**：检测文件变化并触发刷新

## 技术栈建议

| 层级 | 技术选型 |
|------|----------|
| 前端框架 | React 18 + TypeScript (与 ai-ui-runtime 保持一致) |
| 状态管理 | Zustand / Redux Toolkit |
| 终端集成 | xterm.js + node-pty |
| 构建工具 | Vite (快速开发) / Webpack |
| UI 组件库 | Ant Design / Material UI |
| 文件监听 | chokidar |
| **后端服务** | **Node.js + Express/Fastify** |
| **应用架构** | **纯浏览器 + 本地 Node.js 服务** |
| **终端** | **xterm.js + node-pty (支持任意 CLI 工具)** |

## 关键交互流程

```
用户操作                                        系统响应
  │                                               │
  ├─→ 在文件树中选择页面 ──────────────────────→ 渲染区加载对应 HTML
  │                                               │
  ├─→ 点击渲染区元素 ──────────────────────────→ 高亮选中 + 生成组件线索
  │                                               │
  ├─→ 拖拽/缩放元素 ───────────────────────────→ 视觉预览 + 生成 Intent
  │                                               │
  ├─→ 在对话框输入修改指令 ────────────────────→ 绑定 Intent 上下文
  │                                               │
  ├─→ 发送对话请求 ────────────────────────────→ 调用 AI Code CLI
  │                                               │
  ├─→ CLI 执行文件修改 ──────────────────────────→ 文件监听触发刷新
  │                                               │
  └─→ 渲染区自动更新 ──────────────────────────→ 展示修改后效果
```

## 与 ai-ui-runtime 的集成策略

### 复用组件
- `packages/shared/types.ts` - 类型定义
- `packages/ui-runtime/domScanner.ts` - DOM 扫描
- `packages/intent-engine/intentEngine.ts` - 意图生成
- `packages/browser-extension/src/overlay/selection.ts` - 选择逻辑
- `packages/browser-extension/src/overlay/drag.ts` - 拖拽绑定
- `packages/browser-extension/src/overlay/resize.ts` - 缩放绑定

### 需要调整
- 移除 Chrome Extension 消息通信层
- 将 Shadow DOM 挂载改为直接嵌入渲染区
- 适配 iframe 跨域通信（如需要）
- 简化"复制给 AI"流程，改为直接调用 CLI

## 风险与挑战

| 风险 | 影响 | 缓解策略 |
|------|------|----------|
| iframe 跨域限制 | 中 | 使用同源策略，所有渲染文件在本地服务 |
| 终端集成复杂度 | 中 | 优先使用成熟方案（xterm.js + node-pty） |
| AI CLI 响应延迟 | 低 | 添加加载状态、乐观更新 |
| 文件冲突 | 中 | 实现文件锁/版本管理 |
| 性能问题 | 低 | 虚拟滚动、懒加载、按需渲染 |
